警惕一种常见的 Web 应用程序漏洞,它会使数据面临风险。美国和澳大利亚的网络安全机构警告称,有一种称为不安全直接对象引用 (IDOR) 漏洞的常见安全漏洞,恶意行为者利用它来篡改 Web 应用程序数据。这种访问控制缺陷会导致 Web 应用程序执行不足的身份验证和授权检查。当成功利用时,IDOR 漏洞允许黑客通过使用合法用户的标识符向网站或 Web API 发送请求来修改、删除和访问数据。澳大利亚网络安全中心 (ACSC)、美国网络安全和基础设施安全局 (CISA) 和美国国家安全局 (NSA) 在联合公告中写道:“这些漏洞经常被恶意行为者在数据泄露事件中利用,因为它们很常见,在开发过程之外很难预防,并且可以大规模滥用。”该文件描述了 IDOR 漏洞,解释了它们可能被利用的方式,并详细说明了针对 Web 应用程序供应商、设计师、开发者和最终用户的缓解措施。推荐的缓解措施包括:实施默认安全和设计安全原则,确保 Web 应用程序对每个修改、删除和访问数据的请求进行身份验证和授权检查,使用自动化代码审查工具发现并修复 IDOR 漏洞,不在 URL 上暴露 ID、名称和密钥,而是用密码学上强健的随机值替换它们。对于最终用户,建议进行安全尽职调查,尽快修补 Web 应用程序,并对其 Web 应用程序进行漏洞扫描和渗透测试。

S3对象存储无法保护你免受的是你自己。让我们一起看看我们需要防范的三类风险。(1) 数据丢失的第一大原因是人为错误、意外删除或用垃圾数据覆盖对象。这是你想避免的情况。(2) 第二类风险与不可预测的事件有关:软件问题、硬件问题(磁盘故障)、数据中心停机或自然灾害/人为灾难。(3) 第三类风险是让安全专家夜不能寐的东西——恶意行为:恶意软件、勒索软件和病毒、破坏行为、DDoS......
安全非常重要且不容商量,让我们来看看5条简单规则来防范这些风险。
规则1 – 版本控制
版本控制有助于防止意外覆盖。您可以在意外删除后恢复一个版本,或在数据损坏时检索特定版本。
规则2 – 不可变性
当您的主要存储系统必须开放和可用时,您的备份数据应被隔离且不可变。使用S3对象锁定API实现一次写入多次读取(WORM)模型。您可以根据需求、业务和数据类型定义不同的参数:保留期、法律模式、治理模式、合规模式。此规则帮助您的组织遵守合规要求。为了在法律限定的时间内保留日志,“合规模式”将帮助您设置所需的时间段。这非常方便,因为日志每秒都在生成,因此很难跟踪。借助合规模式,您不再需要担心这一点,您可以设置1年、3年或5年的期限,日志将在指定期间内受到保护。
规则3 – 异地数据复制
为了防止硬件故障或地理事件的影响,请遵循著名的3+2+1模型。在设置数据副本之前,您需要评估目标的RPO和RTO。RPO(恢复点目标)= 在发生地理故障的情况下,什么时间点的数据快照对您来说是可以接受的,以便在恢复数据时损失最少的数据;RTO(恢复时间目标)= 在发生地理故障的情况下,恢复数据的可接受时间是多少。当然,每个人都希望实现零秒恢复,但这是否必要?这样的恢复计划需要昂贵的资源来维护。一个好建议是按重要性类别对数据进行分类,并根据类别调整此计划并实施备份保留策略。
规则4 – 加密
当您的数据不使用时,可以使用自己的密钥对其进行加密。我们使用基于AES-256协议的功能。加密您的数据:使用您自己的密钥和基于AES-256的加密。请注意,在传输过程中,数据通过TLS协议进行了加密。
规则5 – 用户策略
仅授予执行任务所需的权限,使用用户策略存储桶策略(即将推出)。每月提取您的S3策略并检查它们;这不会花费太多时间并且可以自动化。确保您知道所有用户,并且权限适合每个角色。永远不要让通配符*提供对敏感存储桶/对象的访问。
